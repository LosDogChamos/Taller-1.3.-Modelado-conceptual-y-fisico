📘 Introducción

El modelado de bases de datos es la columna vertebral de cualquier sistema de información robusto. Este taller (versión 1.3) aborda la transición desde el modelo conceptual (entidades, relaciones, restricciones semánticas) hacia el modelo físico (sentencias DDL, tipos de datos, claves, índices). A partir de dos escenarios de negocio —un sistema de inventario con control de entradas/salidas y una plataforma de comercio electrónico— el equipo Los Dog Chamos 2.0 desarrolló los diagramas y su correspondiente implementación en SQL.

🔍 Análisis y decisiones de diseño

  • Entidad DETALLE actúa como tabla de unión entre ORDEN_COMPRA y ELECTRODOMÉSTICO, pero a su vez se relaciona con COMPROBANTE para registrar salidas. Para cumplir el requerimiento de signo, la cantidad acepta valores positivos (entrada) y negativos (salida).
  • PROVEEDOR (1) → (N) ORDEN_COMPRA: un proveedor puede tener muchas órdenes; cada orden pertenece a un único proveedor.
  • CLIENTE y COMPROBANTE (1:N): un cliente puede tener varios comprobantes (ventas).
  • El detalle hereda las claves foráneas de electrodoméstico, orden y comprobante (opcional según sea entrada/salida).

📄 Script de implementación física (DDL reducido)

-- Extracto del modelo físico (BDLavanderia) CREATE TABLE proveedor ( id_proveedor INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100) NOT NULL, ruc VARCHAR(20) UNIQUE, telefono VARCHAR(20) ); CREATE TABLE orden_compra ( id_orden INT PRIMARY KEY AUTO_INCREMENT, fecha DATE NOT NULL, hora TIME, monto_total DECIMAL(10,2), id_proveedor INT, FOREIGN KEY (id_proveedor) REFERENCES proveedor(id_proveedor) ); CREATE TABLE electrodomestico ( id_electro INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100), descripcion TEXT, marca VARCHAR(50), precio_compra DECIMAL(10,2), precio_venta DECIMAL(10,2) ); CREATE TABLE comprobante ( id_comprobante INT PRIMARY KEY AUTO_INCREMENT, serie VARCHAR(20), fecha_emision DATETIME, id_cliente INT ); CREATE TABLE detalle_movimiento ( id_detalle INT PRIMARY KEY AUTO_INCREMENT, cantidad INT, subtotal DECIMAL(10,2), id_electro INT, id_orden INT NULL, id_comprobante INT NULL, FOREIGN KEY (id_electro) REFERENCES electrodomestico(id_electro), FOREIGN KEY (id_orden) REFERENCES orden_compra(id_orden), FOREIGN KEY (id_comprobante) REFERENCES comprobante(id_comprobante), CHECK ( (id_orden IS NOT NULL AND id_comprobante IS NULL) OR (id_orden IS NULL AND id_comprobante IS NOT NULL) ) );

🧩 Detalles del modelado

  • Producto pertenece exactamente a una categoría (relación N:1) y a una marca (N:1).
  • Cliente - Dirección: 1:N (un cliente puede tener múltiples direcciones de envío).
  • Pedido contiene varios ItemPedido (detalle), y cada item referencia un producto.
  • Cliente puede opinar sobre Producto: la entidad OPINION tiene doble referencia a cliente y producto.

📄 Script físico (fragmento)

-- Tablas para comercio electrónico (BD_Ecommerce) CREATE TABLE cliente ( id_cliente INT PRIMARY KEY AUTO_INCREMENT, nombre_completo VARCHAR(120), email VARCHAR(150) UNIQUE NOT NULL, contrasena VARCHAR(255) ); CREATE TABLE direccion_envio ( id_direccion INT PRIMARY KEY AUTO_INCREMENT, direccion TEXT, ciudad VARCHAR(60), id_cliente INT, FOREIGN KEY (id_cliente) REFERENCES cliente(id_cliente) ); CREATE TABLE marca ( id_marca INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(60), pais_origen VARCHAR(60) ); CREATE TABLE categoria ( id_categoria INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(60), descripcion TEXT ); CREATE TABLE producto ( id_producto INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100), descripcion TEXT, precio DECIMAL(10,2), stock INT, id_marca INT, id_categoria INT, FOREIGN KEY (id_marca) REFERENCES marca(id_marca), FOREIGN KEY (id_categoria) REFERENCES categoria(id_categoria) ); CREATE TABLE pedido ( id_pedido INT PRIMARY KEY AUTO_INCREMENT, fecha DATETIME, estado ENUM('Pendiente','Enviado','Entregado'), id_cliente INT, FOREIGN KEY (id_cliente) REFERENCES cliente(id_cliente) ); CREATE TABLE item_pedido ( id_item INT PRIMARY KEY AUTO_INCREMENT, cantidad INT, id_pedido INT, id_producto INT, FOREIGN KEY (id_pedido) REFERENCES pedido(id_pedido), FOREIGN KEY (id_producto) REFERENCES producto(id_producto) ); CREATE TABLE opinion ( id_opinion INT PRIMARY KEY AUTO_INCREMENT, calificacion TINYINT CHECK (calificacion BETWEEN 1 AND 5), comentario TEXT, fecha DATE, id_cliente INT, id_producto INT, FOREIGN KEY (id_cliente) REFERENCES cliente(id_cliente), FOREIGN KEY (id_producto) REFERENCES producto(id_producto) );

✅ Reflexiones finales

  • Modelado conceptual vs físico: las imágenes 1.0 y 2.0 (Lucidchart) muestran la vista conceptual; las imágenes 1.1 y 2.1 (SQL) representan la implementación física.
  • Manejo de signos en inventario: la solución con un único detalle simplifica el control de stock.
  • Integridad referencial: todas las relaciones están protegidas con FOREIGN KEY.
  • Direcciones múltiples: se normalizó en tabla separada para permitir N direcciones por cliente.
